## 1. 产品概述与愿景

### 1.1 产品定位

东软智慧云脑诊疗平台是一套面向高校教学实训的**三端智能门诊诊疗原型系统**，采用 Spring Boot 3 + Vue 3 前后端分离架构，融合真实大模型 API、本地处方规则库与传统 HIS 门诊业务流程，覆盖"诊前分诊—线上挂号—医生接诊—病历生成—处方审核—患者查看"的一期诊疗闭环。

平台定位为**教学实训产品 + 可演示产品原型**。目标不是生产级医疗系统，不直接替代医生诊断和处方决策，而是让全栈开发学习者通过动手实践，理解传统业务模块如何与 AI 能力有机结合、前后端如何通过 RESTful API 协作，以及真实医疗流程中的角色、排班、号源、药品规则和人工确认机制如何落到系统设计中。

### 1.2 产品愿景

让每一位学生通过本项目，完整理解以下核心命题：

- **前后端分离架构**下，Spring Boot 3（后端）与 Vue 3（前端）分别承担什么职责，如何通过 RESTful API + JWT 认证协作。
- **传统医疗业务**（患者管理、药品管理、挂号、处方、病历）如何从数据库设计到前端交互的完整链路。
- **AI 能力融合**：智能分诊、AI 病历生成、"本地规则 + 大模型解释"处方审核如何作为独立 Service 嵌入传统业务流程，分别扮演"智能入口""人机协作""安全副驾驶"角色。AI 诊疗建议、数据看板与评价反馈作为 P1 加分能力，不阻塞一期核心交付。
- **数据驱动闭环**：通过就诊评价、AI 调用记录与处方审核记录，支撑后续数据可视化和诊疗质量反馈。
- **全栈联调**：如何处理跨域、异步状态、错误边界、AI 服务降级等真实工程问题。

### 1.3 核心价值主张

| 维度             | 价值                                                                                                    |
| ---------------- | ------------------------------------------------------------------------------------------------------- |
| **患者端** | 手机窗口式体验：输入症状 → AI 推荐科室医生 → 一键挂号 → 查看处方病历，形成患者侧在线就诊入口         |
| **医生端** | PC 工作台体验：查看患者 → 录入问诊 → AI 生成病历 → 选择药品 → 本地规则 + 大模型审核处方 → 安全开方 |
| **管理端** | PC 后台体验：维护科室、医生、排班号源、药品和处方规则，保障挂号与处方审核有可靠基础数据                 |
| **学习者** | 完整的前后端分离 + AI 融合 + 大数据分析开发训练，从零到一交付可运行的全栈应用                           |

## 4. 患者端功能需求

### 4.1 注册与登录

**【功能概述】**

患者通过注册创建账号，登录后获取 JWT 令牌，后续请求自动携带令牌以访问受保护接口。

**【功能要点】**

| ID       | 需求描述                                                              |
| -------- | --------------------------------------------------------------------- |
| P-REG-01 | 注册页面包含：用户名、手机号、密码、确认密码等字段                    |
| P-REG-02 | 前端对密码长度（≥6 位）、手机号格式进行校验                          |
| P-REG-03 | 注册成功后跳转至登录页                                                |
| P-REG-04 | 登录页面包含：用户名/手机号、密码                                     |
| P-REG-05 | 登录成功后后端返回 JWT 令牌，前端存储于 localStorage                  |
| P-REG-06 | Axios 拦截器自动在请求头携带 token：`Authorization: Bearer <token>` |
| P-REG-07 | 登录失败时页面提示错误原因（密码错误/用户不存在）                     |

**【数据模型关联】**

- 患者表（Patient）：存储患者账号与基本信息

### 4.3 智能分诊与挂号

**【功能概述】**

患者输入症状描述后，系统调用 AI 分诊引擎，推荐最合适的科室及医生列表。患者可在推荐结果中选择医生，并基于管理端维护的排班号源完成挂号。

这是 **AI 与传统业务融合的核心场景之一**——AI 作为挂号的智能入口。

**【功能要点】**

| ID       | 需求描述                                                                               |
| -------- | -------------------------------------------------------------------------------------- |
| P-TRI-01 | 智能分诊页面提供多行文本输入框，供患者输入症状描述（如"胸痛伴气短"）                   |
| P-TRI-02 | 点击"智能分诊"按钮后，调用后端 AI 分诊接口                                             |
| P-TRI-03 | 调用期间显示 Loading 加载状态（骨架屏或旋转动画），按钮置灰不可重复点击                |

### 5.3 问诊对话录入与 AI 诊疗建议

该能力属于 **诊疗辅助参考**，不替代医生诊断。为降低一期范围与医疗安全表述风险，AI 诊疗建议不作为 P0 必须交付项。

| D-CON-03 | P0 提供"AI 生成病历"入口，将当前对话内容发送至病历生成引擎                                                              |
| D-CON-04 | P1 可提供"AI 诊疗建议"按钮，AI 返回可能的诊疗建议列表（按置信度排序），每条包含：疾病名称、诊断依据、建议进一步检查项目 |

**【用户操作流程】**

### 7.5 AI 能力降级策略

AI 服务为外部依赖，存在网络延迟、超时、服务不可用等风险。平台需内置降级策略：

| 场景                 | 降级行为                                                                         |
| -------------------- | -------------------------------------------------------------------------------- |
| AI 分诊不可用        | 前端提示"AI 分诊暂时不可用"，展示手动科室选择入口，不阻塞挂号                    |
| AI 诊疗建议不可用    | 前端提示"AI 建议暂不可用"，医生自行判断诊断方向                                  |
| 大模型处方审核不可用 | 本地规则仍然执行，前端提示"大模型解释暂不可用"，医生可查看本地规则结果后继续处理 |
| AI 病历生成不可用    | 前端提示"AI 生成暂不可用"，医生可直接手动填写病历表单                            |
| AI 调用超时          | 设置合理超时时间（建议 15-30 秒），超时后走上述降级路径                          |
| AI 返回格式异常      | 后端解析失败时返回降级响应，前端按不可用处理                                     |

**核心原则**：AI 是辅助，不应成为流程瓶颈。AI 不可用时系统核心流程（挂号、问诊、开方、保存病历）必须保持可用。
